**Step B — Create GitHub Actions workflow:**

Create `.github/workflows/deploy.yml`:

```yaml
name: Deploy to ECS

on:
  push:
    branches: [main]

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - name: Deploy via SSH
        uses: appleboy/ssh-action@v1
        with:
          host: ${{ secrets.ECS_HOST }}
          username: ${{ secrets.ECS_USER }}
          key: ${{ secrets.ECS_SSH_KEY }}
          port: ${{ secrets.ECS_PORT || 22 }}
          script: |
            cd /var/www/bank-weekly-task
            git pull origin main
            npm install --production
            npm run build
            pm2 reload ecosystem.config.cjs --env production || pm2 start ecosystem.config.cjs --env production
```

This workflow:
1. Triggers on push to `main` branch only
2. Uses `appleboy/ssh-action@v1` to SSH into the ECS server
3. Navigates to the project directory on the server
4. Pulls latest code, installs production deps, builds frontend
5. Reloads pm2 (or starts if first deploy)

Required GitHub Secrets (user must configure these in the repo Settings > Secrets):
- `ECS_HOST` — Server IP address or hostname
- `ECS_USER` — SSH username (e.g., `root` or deploy user)
- `ECS_SSH_KEY` — Private SSH key for authentication
- `ECS_PORT` — SSH port (optional, defaults to 22)

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GitHub Actions -> ECS via SSH | CI/CD pipeline connects to production server |
| GitHub Secrets | Stores SSH credentials for deployment |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-09 | Spoofing | SSH deployment | mitigate | SSH key authentication (not password). Key stored in GitHub encrypted secrets, not in repo. Only `main` branch triggers deploy. |
| T-04-10 | Tampering | deploy.yml workflow | mitigate | Workflow file is in the repo — any modification requires a commit. Branch protection on main recommended but not enforced in v1. |
| T-04-11 | Information Disclosure | GitHub Secrets | mitigate | Secrets are encrypted by GitHub, never exposed in logs. SSH key is the minimum credential needed. |
| T-04-12 | Denial of Service | pm2 process | mitigate | pm2 autorestart enabled; max_memory_restart at 256M prevents memory leaks from consuming all server memory. |
| T-04-13 | Elevation of Privilege | SSH user | accept | Using the server's SSH user (likely root for small ECS). For 5-10 user internal tool, acceptable. Production hardening (deploy user, limited permissions) is v2 concern. |
</threat_model>
