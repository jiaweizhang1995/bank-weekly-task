  <name>Task 1: Create auth middleware with token generation and verification</name>
  <files>middleware/auth.js, middleware/auth.test.js</files>
  <read_first>data/store.js, server.js</read_first>
  <behavior>
    - generateToken(identifier) returns a hex string token of length 64
    - Two calls to generateToken produce different tokens
    - verifyToken(token) returns true for a token that was generated and not expired
    - verifyToken("invalid-garbage") returns false
    - verifyToken(expiredToken) returns false after TTL passes
    - requireAdmin middleware calls next() when valid token in Authorization header as "Bearer {token}"
    - requireAdmin middleware returns 401 JSON when no Authorization header
    - requireAdmin middleware returns 401 JSON when token is invalid
  </behavior>
  <action>
Create directory `middleware/` and file `middleware/auth.js` (ES module).

Token strategy: Use Node.js built-in `crypto.randomBytes(32).toString('hex')` to generate 64-char hex tokens. Store active tokens in a module-level Map: `Map<token, { createdAt: number }>`. Token TTL = 24 hours (86400000 ms). No external dependencies needed.

Exports:
1. `generateToken()` — creates random token, stores in Map with timestamp, returns token string
2. `verifyToken(token)` — checks Map for token, checks if (Date.now() - createdAt) < TTL, returns boolean. If expired, deletes from Map.
3. `requireAdmin(req, res, next)` — Express middleware. Reads `req.headers.authorization`, expects format `Bearer {token}`. Extracts token, calls verifyToken(). If valid: `next()`. If invalid or missing: `res.status(401).json({ error: '未授权，请先登录' })`.
4. `clearTokens()` — clears the token Map (for testing only).

Create `middleware/auth.test.js` using Node.js built-in test runner (`node:test` + `node:assert`). Test all behaviors listed above. For expiry test, temporarily override TTL or use a short TTL helper — simplest approach: export a `_setTTL(ms)` function used only in tests, reset after.

Run tests RED first, then implement GREEN.

Endpoints:

1. **POST /login** (no auth required)
   - Body: `{ pin: "string" }`
   - Validate: `pin` exists and is a string. If missing: 400 `{ error: '请提供PIN码' }`
   - Compare `pin` against `getData().adminPin`
   - Match: call `generateToken()`, return 200 `{ token: "..." }`
   - No match: return 401 `{ error: 'PIN码错误' }`

2. **PUT /pin** (requireAdmin middleware)
   - Body: `{ oldPin: "string", newPin: "string" }`
   - Validate: both fields exist and newPin is string length 4-8. If invalid: 400 `{ error: '新PIN码必须为4-8位' }`
   - Verify oldPin matches current `getData().adminPin`. If not: 403 `{ error: '原PIN码错误' }`
   - Update: `const data = getData(); data.adminPin = newPin; saveData(data);`
   - Return 200 `{ message: 'PIN码已更新' }`

Export the router as default export.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client -> /api/admin/login | Untrusted PIN input from network |
| client -> /api/admin/pin | Untrusted PIN change request; requires valid token |
| Authorization header -> requireAdmin | Bearer token from untrusted client |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Spoofing | /api/admin/login | mitigate | Token is 64-char crypto.randomBytes hex — brute force infeasible; 401 on wrong PIN reveals no information beyond "wrong" |
| T-02-02 | Tampering | /api/admin/pin | mitigate | requireAdmin middleware validates token before handler runs; oldPin must match current PIN |
| T-02-03 | Information Disclosure | /api/admin/login | mitigate | Error messages are generic Chinese strings; PIN value never included in responses |
| T-02-04 | Denial of Service | /api/admin/login | accept | No rate limiting in v1; acceptable for 5-10 user team on private network. Rate limiting is a v2 concern. |
| T-02-05 | Elevation of Privilege | requireAdmin middleware | mitigate | Every admin-write route uses requireAdmin middleware; token stored in memory Map with 24h TTL |
</threat_model>
