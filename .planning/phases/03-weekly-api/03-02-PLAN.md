Add the following endpoints to the EXISTING `routes/week.js` (append after the existing PUT /settings route, before `export default router;`):

1. **POST /tasks** (requireAdmin middleware — per WEEK-04)
   - Body: `{ name: "string", desc?: "string" }`
   - Validate: `name` must exist, be a string, and `name.trim().length > 0`. If invalid: 400 `{ error: '请提供任务名称' }`
   - Initialize currentWeek if null: `if (!data.currentWeek) data.currentWeek = { tasks: [], status: {}, penalty: "", deadline: "", announcement: "" };`
   - Create task object: `const task = { id: Date.now().toString(), name: name.trim(), desc: (desc || "").trim() };`
   - Add to tasks array: `data.currentWeek.tasks.push(task); saveData(data);`
   - Return 201 `{ message: '任务已添加', task }`

2. **DELETE /tasks/:taskId** (requireAdmin middleware — per WEEK-05)
   - Get taskId from `req.params.taskId`
   - Initialize currentWeek if null (same pattern)
   - Find task: `const idx = data.currentWeek.tasks.findIndex(t => t.id === taskId);`
   - If idx === -1: return 404 `{ error: '任务不存在' }`
   - Remove task: `data.currentWeek.tasks.splice(idx, 1);`
   - Clean up status entries — remove this taskId from every member's status object:
     ```javascript
     const status = data.currentWeek.status;
     for (const member of Object.keys(status)) {
       if (status[member] && taskId in status[member]) {
         delete status[member][taskId];
       }
     }
     ```
   - `saveData(data);`
   - Return 200 `{ message: '任务已删除' }`

3. **PUT /tasks/:taskId/status/:member** (conditional auth — per STAT-01, STAT-02, STAT-03)
   - Get `taskId` from `req.params.taskId` and `member` from `req.params.member` (URL-decoded automatically by Express)
   - Body: `{ status: "done" | "rejected" | null }`
   - Validate status value: must be one of `"done"`, `"rejected"`, or `null`. If invalid: 400 `{ error: '状态值无效，必须为 done、rejected 或 null' }`
   - Initialize currentWeek if null (same pattern)
   - Validate member exists: `if (!data.members.includes(member))` return 404 `{ error: '成员不存在' }`
   - Validate task exists: `if (!data.currentWeek.tasks.find(t => t.id === taskId))` return 404 `{ error: '任务不存在' }`
   - **Auth logic:**
     - If `status === "done"`: No admin token needed (member self-marking per STAT-01 and AUTH-04). Members identify by name in URL.
     - If `status === "rejected"` or `status === null`: Requires admin token. Check `req.headers.authorization` manually — call `verifyToken()` imported from middleware/auth.js. If no valid token: 401 `{ error: '未授权，请先登录' }`
   - **Important:** Import `verifyToken` from middleware/auth.js (in addition to existing `requireAdmin` import). The import line becomes: `import { requireAdmin, verifyToken } from '../middleware/auth.js';`
   - Update status:
     ```javascript
     if (!data.currentWeek.status[member]) data.currentWeek.status[member] = {};
     data.currentWeek.status[member][taskId] = req.body.status;
     saveData(data);
     ```
   - Return 200 `{ message: '状态已更新', member, taskId, status: req.body.status }`

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client -> POST /api/week/tasks | Admin write — untrusted task name/desc, requires token |
| client -> DELETE /api/week/tasks/:taskId | Admin write — untrusted URL param, requires token |
| client -> PUT /api/week/tasks/:taskId/status/:member | Mixed — "done" is member action (no token), "rejected"/null require admin token |
| client -> POST /api/week/reset | Admin write — destructive operation, requires token |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-07 | Spoofing | POST /api/week/tasks | mitigate | requireAdmin middleware validates Bearer token |
| T-03-08 | Spoofing | PUT /status (member self-mark) | accept | Members identify by name in URL per AUTH-04 design. No auth for "done" status. Team is small (5-10) and trusted. Member name must exist in data.members array. |
| T-03-09 | Spoofing | PUT /status (admin reject/reset) | mitigate | verifyToken called manually for "rejected" and null status values; 401 if no valid token |
| T-03-10 | Tampering | PUT /status body | mitigate | Status value validated against whitelist: only "done", "rejected", or null accepted; 400 on anything else |
| T-03-11 | Tampering | POST /api/week/tasks body | mitigate | Task name validated (non-empty string after trim); desc defaults to empty string; task ID is server-generated Date.now() — not client-provided |
| T-03-12 | Tampering | DELETE /api/week/tasks/:taskId | mitigate | requireAdmin validates token; taskId checked against existing tasks (404 if not found); status cleanup prevents orphaned entries |
| T-03-13 | Denial of Service | POST /api/week/reset | mitigate | requireAdmin middleware prevents unauthorized reset; only admin with valid token can clear weekly data |
| T-03-14 | Information Disclosure | PUT /status response | accept | Response includes member name and task ID — same data visible in public GET /api/week. No additional exposure. |
| T-03-15 | Elevation of Privilege | PUT /status "done" without token | mitigate | Only "done" allowed without token. "rejected" and null require verifyToken check. Member must exist in data.members. Impersonation accepted per AUTH-04 design (small trusted team). |
</threat_model>
